1. Request lifecycle

1

Understand GEN understand.py

The LLM reads the request as JSON — intent (specification / quotation / comparison / concept / search / general), category, and parameters (CFM, dimensions, qty…). A regex layer is the offline fallback.

↓
2

Route DET main._prepare

Completeness decides the policy: enough inputs (or "refer db") → ATS / data mode; otherwise → Consulting / knowledge mode. Quotation intent uses data mode as soon as the sizing driver is present.

↓
3

Retrieve DET retriever.py · ChromaDB

Semantic vector search (MiniLM embeddings) over the knowledge base, category-filtered. Analytics/comparison pull the whole corpus or the named records.

↓
4

Reason — the Resolver DET resolver.py · analysis.py

One engine authors the specification. A policy decides which sources may set a value; engineering rules compute physics-based fields; case-based reasoning interpolates/scales/reuses from your offers. See §2.

↓
5

Validate & score DET validate.py · ledger.py

Every value is sanity-checked against engineering limits and cross-checked; a decision ledger records each value's origin; a multi-factor confidence is computed.

↓
6

Explain GEN llm.py · prompt.py

The LLM narrates the computed result (spec) or answers conversationally (search) — fed the exact decided values, so it cannot drift from them. Small talk & exact counts are answered in code, skipping the LLM.

↓
7

Verify & assemble DET llm._verify · main._meta

Direct fact-lookups are re-checked against the records before display. The response carries the answer, provenance, confidence and cited source files (and a quotation, if requested).

2. Inside the reasoning engine (the Resolver)

One code path, two policies — the difference is only which evidence may author a value:

Provenance originMeaningConsultingATS
givenClient requirement (authoritative)✓✓
ruleEngineering formula + standard✓✓
interpolated / scaledSynthesised/scaled from historical offers—✓
consistentHistorical consensus across offers—✓
reusedTaken from the nearest offer—✓
tbd / estimateDeferred — asks for the input, never invents✓—

Confidence is multi-factor: engineering match %, historical support, rule coverage, missing inputs and validation results. Verified byte-identical by a golden regression suite (10 cases across both policies).

3. What the LLM does — and does not do

Generative AI handles

  • Understanding the request (intent, category, parameters)
  • Writing the engineering narrative for a spec
  • Conversational Q&A and explanations
  • Quotation wording

Deterministic code handles

  • Every specification number (rules + your offers)
  • Every price (nearest offer, driver-scaled, trend-checked)
  • Counts, lists, breakdowns (exact, not "guessed")
  • Provenance, validation, confidence, source citation

This is a grounded / compound AI system — generative AI for language, symbolic engine for the engineering. It is more than plain retrieval-augmented generation (RAG): retrieval feeds a reasoning engine, not just the prompt.

4. Status — what is built

✔ Done (working prototype)

  • Specification engine — rules + case-based reasoning over 33 real offers (10 categories), with provenance, validation, decision ledger and confidence.
  • One resolver, two policies — Consulting Engineer / ATS Engineering Expert, golden-locked byte-identical.
  • Quotation agent — deterministic budgetary pricing (per-unit, driver-scaled, trend cross-checked) + scope + terms; on-screen card and a Vitech-format PDF draft.
  • Deterministic analytics — exact counts / lists / clients computed in code.
  • Grounded Q&A — conversational answers cited to your data, or flagged as general knowledge; direct lookups fact-verified.

5. Next — the data foundation

→ Next phase: automatic extraction + a structured database

Today's 33 offers were hand-entered. Next we ingest all your files automatically:

  • Extract — offer PDFs via table/text parsing (pdfplumber); CAD via native .dxf parsing where possible, OCR-on-image as the fallback (every OCR'd number flagged for review).
  • Structured (normalized) database — clients · documents · given_data · technical_specs · price_items · dimensions · bom_items — every field queryable with unit, source and confidence. Enables "all scrubbers > 1000 CFM", "offers for client X".
  • Human-in-the-loop review — each extraction is confirmed by an engineer before it enters the master database.
  • Materialize & embed — the database is the single source of truth; the reasoning engine and search run on a view built from it (no engine rewrite).
