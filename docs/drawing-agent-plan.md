# Drawing Agent — Design Plan ("Vitech Drawing Studio")

> Status: **PLAN** (not built). Third agent after Engineering (live) and
> Quotation (live). Client framing: a **studio** — a visual, interactive place
> where a requirement becomes a **2D General-Arrangement (GA) drawing** you can
> see, tweak, annotate and export, with the chat driving the canvas.

## 1. What "studio" means here
Not a chat that returns an image. A **split workspace**: conversation on the
left, a **live drawing canvas** on the right. You describe/adjust the equipment
in chat (or from a generated spec), the GA drawing renders on the canvas, and
you pan/zoom, toggle layers (envelope, dimensions, components, title block),
annotate, and export to **PDF / SVG / DXF**. It is the drafting-room analogue of
the Quotation Agent: the *engineering* is already deterministic; the studio makes
it **drawable and shippable**.

## 2. Golden-rule constraint (non-negotiable)
Same rule #2 as everywhere: **every dimension, count and coordinate is
deterministic** — produced by code from the spec engine's geometry, never
invented by the LLM. The model only orchestrates (routes the request, narrates,
labels) and the drawing is generated by a **geometry → vector** engine. An
unknown dimension renders as a **TBD callout**, never a guessed line. This is
why the drawing consumes `_spec_geometry`, not prose.

## 3. What already exists (the foundation — do not rebuild)
- **`main.py::_spec_geometry`** → a machine-readable **numeric mm envelope**
  (`length/width/height`), an `envelope_source` (given/partial/tbd), a `ready`
  flag, and per-dimension `fields` with `resolved`/`tbd` status.
- **`/api/tools/spec`** already returns this `geometry` block alongside the spec.
- **`app/spec_template.py`** marks geometry-kind fields per category; the TBD
  guardrail already distinguishes a real dimension from a gap.
- **Frontend studio shell** is 80% there: the app is already a split
  chat + right-panel workspace with pan-able surfaces, theming, and the Flowise
  chat wiring (`useAgentChat`, `agentUrl(view)`), plus a `drawing` nav item
  (currently `status: "soon"`).
- **Flowise agent pattern** (Engineering/Quotation) — clone-and-specialise is
  the proven path (`quotation-agent-build.py`).

So the Drawing Agent is mostly **(a) a deterministic GA-drawing engine** and
**(b) a canvas UI** — the requirement→geometry pipeline is done.

## 4. Architecture
```
User ⇄ Flowise "Drawing Agent" (llama3.1:8b, BufferMemory)
          │  tools: generate_specification (reuse), generate_drawing (new),
          │         lookup_project, list_projects
          ▼
Backend  POST /api/tools/drawing  (operation_id: generate_drawing)
          │  1. resolve requirement → spec (existing pipeline)
          │  2. spec.geometry → drawing_engine  (NEW, deterministic)
          │  3. return { svg, dxf, sheet, layers, scale, tbd[], bom[] }
          ▼
Frontend Drawing Studio  (split: chat  |  canvas)
          render SVG on <canvas>/inline-SVG, layer toggles, pan/zoom,
          annotations, export (SVG/PDF/DXF).
```
Python owns all geometry; Flowise only orchestrates; React renders.

## 5. The drawing engine (`backend/app/drawing/`) — the valuable core
A category-agnostic **geometry → 2D vector** builder, mirroring the
`engineering/` package split so the client can extend it the same way.
- **`primitives.py`** — a tiny vector model (mm space): `Line`, `Rect`,
  `Circle`, `Dim` (dimension line + witness lines + arrowheads + text),
  `Text`, `Leader`, `Layer`. No external CAD dep.
- **`views.py`** — projects the mm envelope into standard GA views: **plan**,
  **front elevation**, **side elevation** (third-angle), auto-scaled to a sheet.
- **`symbols.py`** — per-category component glyphs placed from the spec's
  technical details (scrubber tower + tank + blower + demister; booth cabin +
  doors + filter bank + AHU; oven chamber + door + heater; dust collector
  hopper + bags/cartridges + fan). Positions are **derived from dimensions**,
  not drawn free-hand.
- **`title_block.py`** — the **Vitech title block** reusing
  `vitech_letterhead.py` constants (company, address, contacts) — same
  stationery as the spec/quote PDFs. Fields: title, client, ref, scale, date,
  drawn/checked, revision, units, **DRAFT** watermark.
- **`sheet.py`** — lays the 3 views + title block + notes onto an A3/A2 sheet,
  chooses scale (1:10 / 1:20 / 1:50 …) to fit, adds a scale bar + north/flow
  arrow. Emits **SVG** (source of truth) and **DXF** (via a minimal DXF writer
  or `ezdxf` if we accept the dep).
- **`drawing_service.py`** — orchestrator: `build_drawing(spec) -> Drawing`
  (svg text, dxf text, layer list, scale, bom, tbd list). One entry point the
  tool calls. Deterministic + unit-testable (byte-stable SVG for golden tests).

**Extension point (same story as the engine):** to support a new category's GA,
add its `symbols.py` glyph layout keyed on its geometry fields — the sheet/
views/title-block/export plumbing is shared.

## 6. Tool endpoint
`POST /api/tools/drawing` → `operation_id: generate_drawing`:
```
in : { "question": "<requirement or 'draw the last spec'>" }
out: { ok, category, scale, sheet_size,
       svg,            # inline SVG (studio renders this)
       dxf,            # DXF text (export)
       layers: [{id,label,on}],
       envelope_mm, ready,
       tbd: ["Tower height — needs engineering input", ...],
       bom: [{item, qty, spec}],           # from the spec scope
       drawing_markdown }                   # human summary for the agent
```
Guard identical to spec/quote: **no named requirement → `need_requirement`**,
never draw from noise. If `geometry.ready` is false, render what is known + a
**TBD schedule** on the sheet (honest, drawable gaps).

## 7. Flowise "Drawing Agent"
Clone the Engineering Agent (`drawing-agent-build.py`), keep tools
`generate_drawing` + `generate_specification` + `lookup_project` +
`list_projects`. Prompt rules (short — heed the ≤~4k-char lesson):
- Real requirement (equipment + size) → `generate_drawing`; then present the
  `drawing_markdown` summary (the **canvas shows the SVG**, so the chat reply is
  a short summary + the TBD list, not the raw SVG).
- "change L to 6 m / add a door / rescale" → re-issue with the merged change.
- Never state a dimension not in the tool output; TBDs stay TBD.
- Technical theory → (optionally) hand off to the Engineering Agent, same as
  Quotation RULE 6.

## 8. Frontend — the Studio
Promote `drawing` to a **live split view** (chat | canvas), not a RoadmapPage:
- **Canvas**: render returned SVG inline; **pan** (drag), **zoom** (wheel/
  buttons/fit), **layer toggles** (envelope / dimensions / components / title
  block / grid), a **TBD badge** list, **scale bar**.
- **Toolbar**: Export **SVG / PDF / DXF**, sheet size (A3/A2), fit-to-view,
  reset, print.
- **Annotations (phase 2)**: user leaders/notes stored per drawing (local
  first, backend later) — never alter engineered geometry, only overlay.
- Reuse the existing shell, theming, `useAgentChat`; add `DrawingStudio.jsx` +
  `useDrawing` hook. The right panel becomes the canvas when `view==='drawing'`.

## 9. Phasing
- **P0 — Engine MVP**: `drawing/` package → plan + front elevation for
  **wet scrubber** and **paint booth** (best geometry today), SVG only, Vitech
  title block, TBD schedule. Golden test with byte-stable SVG. *Highest value,
  no new data.*
- **P1 — Tool + Agent**: `/api/tools/drawing`, clone the Flowise agent, wire the
  studio canvas (render + pan/zoom + layers + SVG export).
- **P2 — Exports + views**: side elevation, DXF export, PDF (reuse fpdf/SVG→PDF),
  A2/A3 scaling, scale bar.
- **P3 — Studio polish**: annotations, revisions, more categories (oven, dust
  collector, conveyor), component BOM callouts, per-drawing save.

## 10. Client inputs needed (unblocks depth)
1. **Standard GA conventions** — sheet sizes, scales, title-block fields,
   layer/line conventions Vitech drafts to.
2. **A few sample GA drawings** per category (scrubber, booth, oven) to match
   symbol layout + annotation style.
3. **DXF requirement?** — is AutoCAD-editable output needed at P1, or is SVG/PDF
   enough first? (decides whether we take the `ezdxf` dependency.)
4. Component placement rules (e.g. blower always top-mounted, tank offset) where
   not derivable from dimensions.

## 11. Risks / notes
- **DXF fidelity**: a hand-rolled DXF is fine for lines/text; real CAD blocks
  need `ezdxf` (pure-Python, acceptable). Decide at P2.
- **SVG→PDF**: keep the SVG the source of truth; render PDF from it (svglib/
  reportlab) or re-emit via fpdf vector calls to stay dependency-light.
- **Keep geometry deterministic**: the moment a dimension isn't known it is a
  TBD callout — the studio must *look* professional with gaps, so the TBD
  schedule is a first-class sheet element, not an error.
- **Prompt discipline**: llama3.1 degrades past ~4k chars (see Quotation Agent) —
  keep the Drawing Agent prompt short; the canvas carries the output, not chat.
